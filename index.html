ADR-001: Use ECS Fargate over EKS for Microservices

Accepted 2025-09-15

Context

Team of 4 engineers needs to deploy 8 containerized microservices. Must choose between ECS Fargate (serverless containers) and EKS (managed Kubernetes).

Options Considered

  • ECS Fargate — serverless, no cluster management, pay-per-task
  • EKS — full Kubernetes, powerful but requires dedicated ops expertise
  • ECS on EC2 — cheaper compute but requires capacity planning

Decision

ECS Fargate. For a small team, the operational overhead of EKS (upgrades, node groups, networking plugins) outweighs its flexibility. Fargate eliminates capacity planning and reduces on-call burden.

Consequences

  • No cluster management — team focuses on application code
  • ~20% higher per-task cost vs EC2, offset by reduced ops hours
  • Limited to ECS ecosystem; migration to EKS possible later if needed

ADR-002: Adopt Terraform over CloudFormation for IaC

Accepted 2025-08-20

Context

Infrastructure spans AWS primarily, with some Cloudflare DNS and GitHub Actions. Need a consistent IaC tool that handles multi-provider resources.

Options Considered

  • Terraform — multi-provider, large ecosystem, state management
  • CloudFormation — AWS-native, no state file to manage, but AWS-only
  • Pulumi — code-first IaC, but smaller community and learning curve for team

Decision

Terraform with S3 + DynamoDB backend for state. HCL is declarative and team-friendly. The multi-provider support means one tool for AWS, Cloudflare, and GitHub resources.

Consequences

  • State file requires careful management (remote backend, locking)
  • Team needs Terraform training — mitigated with modules and code review
  • Unified workflow for all infrastructure, not just AWS

ADR-003: Implement Blue-Green over Canary Deployments

Accepted 2025-10-05

Context

Production deployments occasionally cause regressions. Need a zero-downtime strategy with instant rollback capability. Running on ECS Fargate behind ALB.

Options Considered

  • Blue-Green — ALB target group swap, instant full cutover and rollback
  • Canary — gradual traffic shift (1% → 5% → 25% → 100%), requires metrics-based automation
  • Rolling — in-place updates, simpler but slower rollback

Decision

Blue-Green via CodeDeploy + ALB. Rollback is a single target-group switch (under 30s). Canary would be ideal but requires more sophisticated monitoring and automated rollback triggers that our current observability stack doesn't support yet.

Consequences

  • Double the task count during deployments (brief cost increase)
  • Instant rollback — critical for customer-facing services
  • Can evolve to canary later once monitoring matures

ADR-004: PostgreSQL on RDS over DynamoDB for Primary Datastore

Superseded 2025-07-10

Context

Application has complex relational data (orders, users, inventory) with joins and transactions. Evaluating managed database options on AWS.

Options Considered

  • RDS PostgreSQL — relational, Multi-AZ, mature ecosystem
  • DynamoDB — serverless NoSQL, single-digit ms latency, but forces denormalized design
  • Aurora Serverless v2 — auto-scaling, but higher cost at steady-state

Decision

RDS PostgreSQL Multi-AZ (db.r6g.large). Relational model fits the domain. Read replicas handle reporting queries. Superseded by ADR-007 which migrates hot-path queries to DynamoDB while keeping PostgreSQL as source of truth.

Consequences

  • Predictable costs with reserved instances
  • Need to manage connection pooling (PgBouncer) at scale
  • Backup/restore well-understood; PITR available
