AWS 3-Tier
Architecture

A production-grade 3-tier architecture on AWS — highly available, auto-scaling, and cost-optimized. This guide walks through every layer from the CDN edge to the database with full configuration.

CloudFront ALB EC2 Auto Scaling RDS Multi-AZ ElastiCache S3 VPC CloudWatch
Overview

Architecture Overview

The 3-tier architecture separates concerns into three distinct layers: Presentation (CDN + static assets), Application (web servers + API), and Data (database + cache). Each tier scales independently and fails independently.

Tier 1 — Web

CloudFront + S3 for static assets. ALB routes dynamic traffic.

Tier 2 — App

EC2 Auto Scaling Group across 2 AZs. Private subnet, no public IPs.

Tier 3 — Data

RDS Multi-AZ primary + read replica. ElastiCache for session and query caching.

Cost Note

This architecture reduced monthly AWS spend from $44K to $26K (41% reduction) at Optima Apps by right-sizing instances, adding ElastiCache to reduce RDS load, and moving static assets to S3 + CloudFront.

Build Guide

01

Create the VPC and Subnets

Create a VPC with a /16 CIDR block. You need six subnets — two public (ALB), two private-app (EC2), two private-data (RDS) — spread across two availability zones.

# VPC
aws ec2 create-vpc --cidr-block 10.0.0.0/16

# Public subnets (for ALB)
aws ec2 create-subnet --vpc-id vpc-xxx --cidr-block 10.0.1.0/24 --availability-zone us-east-1a
aws ec2 create-subnet --vpc-id vpc-xxx --cidr-block 10.0.2.0/24 --availability-zone us-east-1b

# Private app subnets (for EC2)
aws ec2 create-subnet --vpc-id vpc-xxx --cidr-block 10.0.11.0/24 --availability-zone us-east-1a
aws ec2 create-subnet --vpc-id vpc-xxx --cidr-block 10.0.12.0/24 --availability-zone us-east-1b

# Private data subnets (for RDS)
aws ec2 create-subnet --vpc-id vpc-xxx --cidr-block 10.0.21.0/24 --availability-zone us-east-1a
aws ec2 create-subnet --vpc-id vpc-xxx --cidr-block 10.0.22.0/24 --availability-zone us-east-1b
02

Internet Gateway and NAT Gateway

Attach an Internet Gateway to the VPC for public subnet traffic. Add a NAT Gateway in each public subnet so private EC2 instances can reach the internet for updates without being publicly accessible.

# Internet Gateway
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway --vpc-id vpc-xxx --internet-gateway-id igw-xxx

# NAT Gateway in each AZ (requires an Elastic IP each)
aws ec2 allocate-address --domain vpc
aws ec2 create-nat-gateway --subnet-id subnet-public-1a --allocation-id eipalloc-xxx
Cost Warning

NAT Gateways cost ~$32/month each plus data transfer fees. For dev environments use a NAT Instance instead — a t3.micro costs ~$8/month and works the same way.

03

Security Groups

Create four security groups with the principle of least privilege — each layer only accepts traffic from the layer above it.

# ALB security group — accept HTTPS from internet
aws ec2 create-security-group --group-name sg-alb --description "ALB SG" --vpc-id vpc-xxx
aws ec2 authorize-security-group-ingress --group-id sg-alb-id --protocol tcp --port 443 --cidr 0.0.0.0/0
aws ec2 authorize-security-group-ingress --group-id sg-alb-id --protocol tcp --port 80 --cidr 0.0.0.0/0

# EC2 App security group — accept traffic from ALB only
aws ec2 create-security-group --group-name sg-app --description "App SG" --vpc-id vpc-xxx
aws ec2 authorize-security-group-ingress --group-id sg-app-id --protocol tcp --port 8080 --source-group sg-alb-id

# RDS security group — accept traffic from EC2 only
aws ec2 create-security-group --group-name sg-rds --description "RDS SG" --vpc-id vpc-xxx
aws ec2 authorize-security-group-ingress --group-id sg-rds-id --protocol tcp --port 5432 --source-group sg-app-id
04

Application Load Balancer

Deploy the ALB in the public subnets across both AZs. Configure an HTTPS listener with an ACM certificate. Redirect all HTTP traffic to HTTPS.

# Create ALB
aws elbv2 create-load-balancer \
  --name prod-alb \
  --subnets subnet-public-1a subnet-public-1b \
  --security-groups sg-alb-id \
  --scheme internet-facing \
  --type application

# Create target group
aws elbv2 create-target-group \
  --name prod-app-targets \
  --protocol HTTP \
  --port 8080 \
  --vpc-id vpc-xxx \
  --health-check-path /health \
  --health-check-interval-seconds 30

# HTTPS listener (requires ACM cert)
aws elbv2 create-listener \
  --load-balancer-arn arn:aws:elasticloadbalancing:... \
  --protocol HTTPS --port 443 \
  --certificates CertificateArn=arn:aws:acm:... \
  --default-actions Type=forward,TargetGroupArn=arn:aws:...
05

EC2 Auto Scaling Group

Create a Launch Template and Auto Scaling Group in the private app subnets. Set minimum 2 instances (one per AZ), desired 2, max 10. Add a CPU-based scaling policy.

# Launch Template
aws ec2 create-launch-template \
  --launch-template-name prod-app-template \
  --launch-template-data '{
    "ImageId": "ami-0abcdef1234567890",
    "InstanceType": "t3.medium",
    "SecurityGroupIds": ["sg-app-id"],
    "UserData": "base64-encoded-startup-script",
    "IamInstanceProfile": {"Name": "prod-app-role"}
  }'

# Auto Scaling Group
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name prod-asg \
  --launch-template LaunchTemplateName=prod-app-template,Version='$Latest' \
  --min-size 2 --desired-capacity 2 --max-size 10 \
  --vpc-zone-identifier "subnet-app-1a,subnet-app-1b" \
  --target-group-arns arn:aws:elasticloadbalancing:...

# CPU scaling policy — scale up at 70%
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name prod-asg \
  --policy-name cpu-scale-out \
  --policy-type TargetTrackingScaling \
  --target-tracking-configuration '{"PredefinedMetricSpecification":{"PredefinedMetricType":"ASGAverageCPUUtilization"},"TargetValue":70.0}'
06

RDS Multi-AZ PostgreSQL

Deploy RDS in the private data subnets with Multi-AZ enabled for automatic failover. Add a read replica to offload reporting queries from the primary instance.

# Subnet group for RDS
aws rds create-db-subnet-group \
  --db-subnet-group-name prod-db-subnet-group \
  --db-subnet-group-description "Production DB subnets" \
  --subnet-ids subnet-data-1a subnet-data-1b

# RDS Multi-AZ instance
aws rds create-db-instance \
  --db-instance-identifier prod-postgres \
  --db-instance-class db.t3.medium \
  --engine postgres --engine-version 15.4 \
  --master-username prodadmin \
  --master-user-password "use-secrets-manager" \
  --allocated-storage 100 \
  --storage-type gp3 \
  --multi-az \
  --db-subnet-group-name prod-db-subnet-group \
  --vpc-security-group-ids sg-rds-id \
  --backup-retention-period 7 \
  --deletion-protection

# Read replica for reporting
aws rds create-db-instance-read-replica \
  --db-instance-identifier prod-postgres-replica \
  --source-db-instance-identifier prod-postgres
Security

Never pass the master password as a CLI argument in production. Use AWS Secrets Manager and reference the secret ARN in your application.

07

ElastiCache (Redis)

Deploy a Redis cluster in the private subnets for session storage and query caching. This is the single highest-impact change for reducing RDS load — a properly configured cache can reduce database queries by 60–80%.

# ElastiCache subnet group
aws elasticache create-cache-subnet-group \
  --cache-subnet-group-name prod-cache-subnet \
  --cache-subnet-group-description "Production cache subnets" \
  --subnet-ids subnet-data-1a subnet-data-1b

# Redis replication group (Multi-AZ)
aws elasticache create-replication-group \
  --replication-group-id prod-redis \
  --description "Production Redis cluster" \
  --cache-node-type cache.t3.medium \
  --engine redis --engine-version 7.0 \
  --num-cache-clusters 2 \
  --automatic-failover-enabled \
  --cache-subnet-group-name prod-cache-subnet \
  --security-group-ids sg-cache-id
08

CloudFront + S3 for Static Assets

Serve all static files (JS, CSS, images) from S3 through CloudFront. This removes significant load from your EC2 instances and puts assets on the CDN edge closest to the user.

# Create S3 bucket for static assets
aws s3api create-bucket --bucket prod-static-assets --region us-east-1
aws s3api put-bucket-policy --bucket prod-static-assets --policy file://bucket-policy.json

# CloudFront distribution
aws cloudfront create-distribution --distribution-config '{
  "Origins": {
    "Quantity": 1,
    "Items": [{
      "Id": "S3-static",
      "DomainName": "prod-static-assets.s3.amazonaws.com",
      "S3OriginConfig": {"OriginAccessIdentity": "origin-access-identity/cloudfront/xxx"}
    }]
  },
  "DefaultCacheBehavior": {
    "ViewerProtocolPolicy": "redirect-to-https",
    "CachePolicyId": "658327ea-f89d-4fab-a63d-7e88639e58f6"
  },
  "Enabled": true,
  "HttpVersion": "http2"
}'
09

CloudWatch Monitoring

Set up dashboards and alarms for the key metrics that matter: ALB response time, EC2 CPU, RDS connections, and ElastiCache hit rate.

# ALB 5xx error alarm
aws cloudwatch put-metric-alarm \
  --alarm-name "prod-alb-5xx" \
  --metric-name HTTPCode_ELB_5XX_Count \
  --namespace AWS/ApplicationELB \
  --statistic Sum --period 60 \
  --evaluation-periods 2 --threshold 10 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:xxx:prod-alerts

# RDS high connections alarm
aws cloudwatch put-metric-alarm \
  --alarm-name "prod-rds-connections" \
  --metric-name DatabaseConnections \
  --namespace AWS/RDS \
  --statistic Average --period 300 \
  --evaluation-periods 2 --threshold 80 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:us-east-1:xxx:prod-alerts
