Overview

CI/CD Pipeline Architecture

The pipeline moves from source code to production in five automated stages. A developer pushes to main, Jenkins picks it up, and 15 minutes later the new version is live behind a rolling update — no downtime, automatic rollback on failure.

Maven Build

Compile, unit test, integration test with Testcontainers.

Docker Build

Multi-stage build. JDK for compile, JRE for runtime. Minimal image.

Harbor Push

Image tagged with build number, scanned for CVEs, pushed to private registry.

Helm Deploy

Helm chart templates Kubernetes manifests. Rolling update replaces pods one at a time.

HPA + Metrics

Horizontal Pod Autoscaler scales on CPU. Prometheus scrapes Spring Actuator.

Build Guide

01

Multi-Stage Dockerfile

The multi-stage build uses a full JDK image to compile and a minimal JRE image to run. This cuts image size from ~500MB to ~150MB and removes build tools from the production image.

# Stage 1 — Build (JDK + Maven)
FROM eclipse-temurin:17-jdk-alpine AS build
WORKDIR /app

# Cache dependencies separately from source code
COPY pom.xml .
RUN mvn dependency:go-offline -q

# Build the JAR
COPY src ./src
RUN mvn clean package -DskipTests -q

# Stage 2 — Runtime (JRE only — no compiler, no Maven)
FROM eclipse-temurin:17-jre-alpine

# Non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
WORKDIR /app

COPY --from=build /app/target/*.jar app.jar

# JVM flags: use G1GC, limit heap to 75% of container memory
ENV JAVA_OPTS="-XX:+UseG1GC -XX:MaxRAMPercentage=75.0 -XX:+UseStringDeduplication"

USER appuser
EXPOSE 8080

HEALTHCHECK --interval=30s --timeout=5s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS -jar app.jar"]
02

Jenkins Pipeline — Build, Test, Push

The Jenkinsfile defines the full pipeline. Parallel stages build all three services simultaneously to cut total build time from 15 minutes to 5 minutes.

pipeline {
  agent any
  environment {
    HARBOR = 'harbor.company.internal'
    HARBOR_CREDS = credentials('harbor-robot')
    NAMESPACE = 'production'
  }

  stages {
    stage('Build & Test (Parallel)') {
      parallel {
        stage('user-service') {
          steps {
            dir('user-service') {
              sh 'mvn clean verify -B'// includes unit + integration tests
            }
          }
          post { always { junit 'user-service/target/surefire-reports/*.xml' } }
        }
        stage('order-service') {
          steps {
            dir('order-service') { sh 'mvn clean verify -B' }
          }
          post { always { junit 'order-service/target/surefire-reports/*.xml' } }
        }
        stage('payment-service') {
          steps {
            dir('payment-service') { sh 'mvn clean verify -B' }
          }
          post { always { junit 'payment-service/target/surefire-reports/*.xml' } }
        }
      }
    }

    stage('Docker Build & Push') {
      steps {
        script {
          sh "echo ${HARBOR_CREDS_PSW} | docker login ${HARBOR} -u ${HARBOR_CREDS_USR} --password-stdin"
          ['user-service', 'order-service', 'payment-service'].each { svc ->
            sh """
              docker build -t ${HARBOR}/production/${svc}:${BUILD_NUMBER} ${svc}/
              docker push ${HARBOR}/production/${svc}:${BUILD_NUMBER}
            """
          }
        }
      }
    }

    stage('Deploy to Kubernetes') {
      steps {
        script {
          ['user-service', 'order-service', 'payment-service'].each { svc ->
            sh """
              helm upgrade --install ${svc} ./helm/microservice \
                --namespace ${NAMESPACE} \
                --set image.tag=${BUILD_NUMBER} \
                --set image.repository=${HARBOR}/production/${svc} \
                --wait --timeout 5m
            """
          }
        }
      }
    }

    stage('Smoke Test') {
      steps {
        sh '''
          kubectl rollout status deployment/user-service -n production
          kubectl rollout status deployment/order-service -n production
          kubectl rollout status deployment/payment-service -n production
        '''
      }
    }
  }

  post {
    failure {
      // Auto-rollback on pipeline failure
      sh '''
        helm rollback user-service -n production || true
        helm rollback order-service -n production || true
        helm rollback payment-service -n production || true
      '''
    }
  }
}
03

Helm Chart — Kubernetes Manifests

A single reusable Helm chart templates Deployments, Services, Ingress, and HPA for all three services. Values files per service override the defaults.

# helm/microservice/values.yaml — defaults
replicaCount: 2
image:
  repository: harbor.company.internal/production/user-service
  tag: latest
  pullPolicy: IfNotPresent

resources:
  requests:
    cpu: 250m
    memory: 512Mi
  limits:
    cpu: 1000m
    memory: 1Gi

autoscaling:
  enabled: true
  minReplicas: 2
  maxReplicas: 10
  targetCPUUtilizationPercentage: 70

livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 60
  periodSeconds: 15

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  initialDelaySeconds: 30
  periodSeconds: 10

# Rolling update strategy — replace one pod at a time
strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0  # Zero downtime
# helm/microservice/templates/deployment.yaml (key section)
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: {{ .Values.strategy.rollingUpdate.maxSurge }}
      maxUnavailable: {{ .Values.strategy.rollingUpdate.maxUnavailable }}
  template:
    spec:
      containers:
        - name: {{ .Release.Name }}
          image: "{{ .Values.image.repository }}:{{ .Values.image.tag }}"
          resources:
            {{- toYaml .Values.resources | nindent 12 }}
          livenessProbe:
            {{- toYaml .Values.livenessProbe | nindent 12 }}
          readinessProbe:
            {{- toYaml .Values.readinessProbe | nindent 12 }}
          env:
            - name: SPRING_PROFILES_ACTIVE
              value: production
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: {{ .Release.Name }}-db-secret
                  key: password
04

Horizontal Pod Autoscaler

HPA automatically scales pod count based on CPU and memory utilization. When CPU exceeds 70%, Kubernetes adds pods. When it drops, it removes them after a cool-down period.

# helm/microservice/templates/hpa.yaml
{{- if .Values.autoscaling.enabled }}
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: {{ .Release.Name }}
  namespace: {{ .Release.Namespace }}
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: {{ .Release.Name }}
  minReplicas: {{ .Values.autoscaling.minReplicas }}
  maxReplicas: {{ .Values.autoscaling.maxReplicas }}
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: {{ .Values.autoscaling.targetCPUUtilizationPercentage }}
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
  behavior:
    scaleDown:
      stabilizationWindowSeconds: 300  # Wait 5min before scaling down
      policies:
        - type: Pods
          value: 1
          periodSeconds: 60  # Remove 1 pod per minute
    scaleUp:
      stabilizationWindowSeconds: 0  # Scale up immediately
      policies:
        - type: Pods
          value: 4
          periodSeconds: 60  # Add up to 4 pods per minute
{{- end }}
Real World Result

At Optima Apps this HPA configuration handled a 10x traffic spike without manual intervention — the cluster scaled from 2 pods to 18 pods in under 3 minutes and scaled back down within 10 minutes of the spike ending.

05

Prometheus Monitoring

Spring Boot Actuator exposes metrics at /actuator/prometheus. A ServiceMonitor tells Prometheus to scrape it automatically. No manual configuration needed.

# ServiceMonitor — tells Prometheus to scrape this service
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: {{ .Release.Name }}
  labels:
    release: prometheus  # Must match Prometheus selector label
spec:
  selector:
    matchLabels:
      app: {{ .Release.Name }}
  endpoints:
    - port: http
      path: /actuator/prometheus
      interval: 15s
      scrapeTimeout: 10s
# Spring Boot — application.yml
management:
  endpoints:
    web:
      exposure:
        include: health,info,prometheus,metrics
  metrics:
    tags:
      application: ${spring.application.name}
      environment: ${spring.profiles.active}
    export:
      prometheus:
        enabled: true
06

Verification and Rollback

After every deployment verify the rollout status and check that pods are healthy. If anything looks wrong, a single Helm command rolls back the entire release.

# Check all deployments rolled out successfully
kubectl rollout status deployment/user-service -n production
kubectl rollout status deployment/order-service -n production
kubectl rollout status deployment/payment-service -n production

# Check pod health
kubectl get pods -n production -l app=user-service

# Check HPA is working
kubectl get hpa -n production

# View resource usage
kubectl top pods -n production

# Check recent events for issues
kubectl describe deployment user-service -n production | grep -A 20 Events

# Rollback to previous release if needed
helm rollback user-service -n production     # Rolls back one version
helm rollback user-service 3 -n production   # Rolls back to specific revision

# View rollout history
helm history user-service -n production
